2 Run-Length Encoding
Run-length encoding (RLE) is a simple form of lossless data compression. It compresses runs of data into (hope- fully) smaller chunks without losing any information. You will encounter RLE files in PA1.
1

A run of data is a repetition of data, e.g. aaaaaaa. This run contains 7 a’s, so rather than store or transmit all seven, RLE compresses this string into just its count and value, 7a, which provides a nice saving in size. If the data being encoded has many long runs, this form of compression can be quite effective. However, if it has many short runs, it can increase the overall size of the data.
# Some more examples of encoded strings.
abbCCC => 1a2b3C
# Non A-z characters are valid
& ((( @ => 1&1 3(1 1@ eeeeeeeeeeeeeeezzz => 15e3z
In some RLE implementations, single character runs are left as-is and not encoded. Q’s encoded complement is 1Q, which doubles the amount of space used. In this assignment, all encoded single characters will be preceded by 1. While inefficient, this makes your decoder slightly easier to implement!
3 Requirements
Your run-length decoder must be capable of parsing and decoding a run-length encoded string consisting of num- bers and arbitrary characters from its command line, decoding the string to its expanded string, and printing the decoded string. It will be invoked on the command line in just one way:
# Decode the given run-length encoded string
./rldecode 3a2b1c
In this example, your decoder should output aaabbc
The only valid invocation of your decoder is a single argument, which is a valid RLE string.
Any other invocation (zero, two, or more arguments; arguments that are not valid RLE strings) is an error,
and your program should print an error message and exit by returning a non-zero value from main(). Any correct invocation that produces a result should exit by returning a zero value from main().
You must implement three functions for this assignment, only one of which is partially defined in the given code (but the other two have function prototypes):
• int main(int argc, char *argv[]): This function is where all C programs start, and argc and argv are how you access the command line arguments. This function is partially defined in the given code.
• bool isnumber(char c): This function accepts a single char as an argument and returns true if it is a valid number and false otherwise. This function is not defined in the given code, but its prototype appears at the top of the file.
• int decode(char *str): This function accepts a run-length encoded string as an argument, prints the de- coded value of that string followed by an ASCII newline, and returns the integer length of the decoded string as a C integer. It must also handle malformed RLE strings, which are any string that does not ad- here to specific encoding method described in this document. When a malformed string is encountered, decode() should cease printing the decoded RLE string, print an error message, and return the integer value -1.
Note that there is a C library function that could perform the role of isnumber() called isdigit(), and several C library functions that can parse the integers in the RLE string into C integers, including atoi() and strtol(). Any assignment using one of these functions will receive no credit. You must implement your isnumber() function by checking the character yourself, and parse integers from the RLE string yourself.
3.1 Integers in RLE Strings
An integer in our RLE encoding can range from 1 to 2147483647 (that is, 1 to 231 − 1), which matches the positive values of the 32-bit int type on our Linux x86-64 platform. You only need to be able to parse positive integers
2

greater than zero, and you do not need to check for or handle overflow (integers larger than 2147483647). The character - should not be treated specially.
In the inputs tested against your program, an integer count will never precede an integer value. That is, you are not expected the handle the case where a run might be composed of integers, like 111111. With the RLE format described in this document, this would encode to 61 which distorts the rest of the encoded string. Integers will occur only as character counts in a correct encoding.

The Unix system has on-line manual pages that are tremendously useful. All functions in the standard library, most commands, and many concepts are documented in these manual pages. The Unix manual pages on your UB CSE virtual machine image are an allowed resource for this assignment, and all other assignments in this course.
Unix man pages can be viewed with the man command at the terminal prompt. The Unix manual is divided into sections, which is important to you because many C library functions also have commands of the same name, and command manual pages will display instead if the section is not specified! Unix commands are in section 1 of the manual, system calls are in section 2, and library functions are in section 3. We will discuss system calls and library functions later, but for now, remember that you may need to specify the section when looking for the documentation of a function. The man command is invoked as man [section] <pagename>, where the section number is an optional argument and the argument <pagename> represents the page you would like to view.
Man page sections are noted as a number in square brackets after a keyword in this document. For example, putchar [3] indicates that the manual page for the putchar() function is found in section 3 of the Unix manual, andyoucanviewitwiththecommandman 3 putchar.

5.5 C Strings
A C String is an array of characters of type char, terminated by a zero byte (that is, an ASCII NUL character). This means that you can examine the individual characters of a C string by iterating the array until you reach a char with the value 0, idiomatically written as '\0'. For reasons that we will explain later, the type of a string is often expressed as char *. The idiomatic way to iterate a string in C is:
char *str = "value";
for (int i = 0; str[i] != '\0'; i++) {
/* In each iteration of this loop, str[i] is the current character under consideration*/
}
The individual characters in a string can be compared to ASCII characters enclosed in single quotes. For example:
if (isnumber(str[0])) { /* Integer char! */
}

5.6 ASCII Characters
Our modern Unix machines, such as the x86-64 Linux of the course virtual machine, mostly use a character encoding called UTF-8. A character encoding is the method of representing language characters as binary bit patterns. Un-accented Latin characters (such as used for the English language), numbers, and simple punctu- ation fall between the binary (integer) values of 0 and 127 in UTF-8, and map directly to an older character set called ASCII, the American Standard Code for Information Interchange. This includes the digits that you will have to parse for this project.
A particular property of the ASCII character set is that the digits occur in order, from 0 to 9. The ASCII value of the character 0 is 48, the character 1 is 49, and so on. This means that the integer value of an ASCII digit can be calculated by subtracting the value '0' from the character in question. For example:
char c = '7';
int value = c - '0'; /* value = 7 */
You may find the ascii [7] man page useful.
